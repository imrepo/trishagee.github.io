The reason I’ve decided to look at the tests first is that we can actually have good quality automated tests that answer many of the questions that were raised in the first post. And even better, provided they are running in some sort of Continuous Integration environment like Team City, they continue to give us confidence in the code well into the future, as long as they are passing.

Another reason to look the tests is that the tests can often be easier to reason about than the production code - they should state the desired functionality clearly, they should be readable, and they should provide insight into which things the author of the code has considered (and possibly hint at areas they have not thought about).

Does this break any existing tests?

This is one of those areas that should be automated, of course. Code that’s under review should, before going through any expensive human-checking review process, go through your standard compile/test cycle. Whether this is done in CI or locally, it will save a lot of time and effort to find those failures before looking further. TeamCity’s remote run feature is particularly useful in these situations.

If there are failing tests, this is the time to have a conversation with the author of the code to decide if the new/amended functionality was:

  1. supposed to make these tests fail: for example, the requirements ask for a change in existing behaviour. In this case, the existing tests need to be altered, or replaced with new ones that assert the correct behaviour.
  2. incorrectly breaking those tests: two possibilities spring to mind - firstly, these tests relied on some side effects that were changed, intentionally or not, as part of the new code. In this case, these existing tests need to be updated (ideally to be less dependent upon undocumented side effects). Secondly, the new code has broken existing functionality. In which case, the new code needs to be re-written to not affect existing functionality.

Note that under no circumstances should newly-breaking tests result in no action - either the tests need changing, or the production code does.

Are there tests for this new/amended code?

When someone submits code for a change or new feature, whether it’s a “serious” enterprise application or an open source project, I expect to see at least one automated test to show what the new code does. In the Java world, this might be a unit test via JUnit, Spock, or similar, or could be an end-to-end test, maybe driven by a tool like Selenium or Robotium.

It would be rare that new code, whether a bug fix or new feature, wouldn’t need a new or updated test to cover it. Even changes for “non-functional” reasons like performance can frequently be proved via a test.

Adding a test for a bug fix or feature is also often a good way to document what the changed code does (with an implication of “why?”). This is particularly useful in projects like open source projects where documentation typically lags behind implementation.

Can I understand the tests?

Having tests that provide adequate coverage is one thing, but, as implied by my previous statement, the tests will ideally also document the behaviour of the code. If I, as the reviewer, can’t understand the tests, they are not meeting this criteria.

Consider the following:

@Test
public void testId() {
    MyEntity obj = new MyEntity(getDs());
    getDs().save(obj);
    assertEquals(FIRST_ID, obj.getMyLongId());

    obj = new MyEntity(getDs());
    getDs().save(obj);
    assertEquals(SECOND_ID, obj.getMyLongId());
}

It’s a fairly simple test, but I’m not entirely sure what it’s testing. Is it testing the save method? Or getMyLongId? And why does it need to do the same thing twice?

The intent behind the test might be clearer as:

@Test
public void shouldIncrementTheEntityIdByOneOnEverySave() {
    // when
    MyEntity entity = new MyEntity(getDs());
    getDs().save(entity);

    // then
    assertEquals(1L, entity.getMyLongId());

    // when
    entity = new MyEntity(getDs());
    getDs().save(entity);

    // then
    assertEquals(2L, entity.getMyLongId());
}

Obviously the specific steps you take in order to clarify a test’s purpose will depend upon your language, libraries, team and personal preferences. But the aim is to demonstrate that these two tests are functionally identical, but by choosing clearer names, inlining constants and even adding comments, an author can make a test more readable by developers other than him- or herself.

In Summary

This post probably makes the code review process sound almost as if it’s in the testing/QA phase, with this emphasis on writing tests and finding edge cases. And in many ways, it is - the aim is to find potential problems with the code before it is integrated into the main code base, while it’s still inexpensive to fix and the context is still in the developer’s head.

The list above is by no means exhaustive, and professional testers can suggest far more ways to check the code. For just some examples read this article Doing Terrible Things To Your Code.

The point to take away is that as a code reviewer, you should be checking that the original developer has put some thought into the ways his or her code could be used, under which conditions it might break, and dealt with edge cases, preferably “documenting” the expected behaviour (both under normal use and exceptional circumstances) with automated tests.

If this seems like a lot more work for a code review than you expected, then think about it this way: it can often be easier to review the tests in a code review than the production code itself. The tests can give you an idea of what the code is supposed to do, and, if they pass, prove that it does just that.

If the reviewer checks for the existence of tests and checks the correctness of the tests, as a team you can have pretty high confidence that the code works. Moreover, if these tests are run regularly in a CI environment, you can see that the code continues to work - they provide automated regression checking. If code reviewers place a high value on having good quality tests for the code they are reviewing, the value of this code review continues long after the reviewer presses the “Approve” button.
