I’ve written a Gradle Plugin that should, theoretically, do the full MongoDB Java Driver release in an automated fashion. In theory, there’s a single command that you can run in order to release the Java driver:

./gradlew release -DreleaseVersion=<release version>

e.g.

./gradlew release -DreleaseVersion=3.0.0

Because the release process integrates with a number of different systems, you have to provide a number of credentials and settings on your local machine in order for the process to work correctly. You’ll need:

~/.gradle/gradle.properties

sonatypeUsername=mongodb sonatypePassword=[find this in the settings file] signing.keyId= signing.secretKeyRingFile= signing.password= github.credentials.username= github.credentials.password=[if you have two factor auth on, this is a github access token]

You’ll also need: ~/.github login=[user name] password=[password]

or, if you’re using two-factor auth, ~/.github oauth=[github access token]

More details on creating a github access token

More info on specific settings for the Java Driver release

So that’s the settings for the whole release process. However, the release gradle task is actually made up of a number of individual steps. Since there are a number of moving parts, and things that could go wrong with each step, these steps can be run individually if necessary. The most important one is publish, all of the others can easily be replaced with a manual process.

The aim of the following documentation is to: - Define what each individual Gradle task is and what it’s supposed to do - State the requirements or dependencies for each individual task - Outline some of the things that could go wrong with the task

The release steps are, in the order in which they should be executed:

clean If running the full release task, a clean will be done first to make sure that everything is cleanly built. This is a standard Java task and has not been changed in any way for the release process.

prepareRelease Before building or pushing, a number of things are done to put the code/files into the correct state:

  1. Update the version number in the appropriate files. In the case of the Java driver, this is build.gradle and Mongo.java. The release versions in these files are stripped of any ‘-SNAPSHOT’ suffix they might have. If there is no suffix, this task should still complete successfully.
  2. Check all changed files into git. You should make sure you don’t have any changes outstanding before running this, as they’ll get checked in as well. The commit comment will be “Release ${releaseVersion}”, e.g. “Release 3.0.0”
  3. Tag this commit with the release number, e.g. “r3.0.0”
  4. Push the changes and the tag to the remote. This seems to push to the default remote, so if you’re working on your own fork, this pushes to your origin.

Note that all of these steps are pretty easy to do manually.

Requirements

Because this task pushes to a git remote, you’ll need github credentials: ~/.gradle/gradle.properties github.credentials.username= github.credentials.password=

Needs a release version defined, it will do a check to make sure you have this. E.g.

./gradlew prepareRelease -DreleaseVersion=3.0.0

The files that need updating as part of the release process are defined in release.gradle, as filesToUpdate.

Ideally, don’t have any code changes in your current branch, as these will be committed.

Code

The behaviour of this task is defined in PrepareReleaseTask.gradle. It uses JGit for the git integration.

Possible errors

  • Problems authenticating to github. [link to github troubleshooting required]
  • If you run this task multiple times, it should not fail, but you will get multiple git commits with the required release number. Make sure that you have the correct release number in the files (Mongo.java and build.gradle) and passed in to the command line.

Rollback

  • Git: hard reset to the commit you were on before performing the release
  • Git: delete the release tag from local and, if necessary, remote repositories

publish This does the magic of pushing the jar files into Sonatype nexus. This should run on it’s own perfectly happily, and can pubish both snapshot jars and full release jars.

  1. Will ensure jar files are built for: sources; javadoc; individual module classes; “uber” jar with the module classes and any classes from modules it depends on (e.g. for mongo-java-driver, include core and bson as well) but excluding any dependency jars.
  2. Ensures the correct pom.xml is created for each of the artifacts
  3. Publishes to the snapshot repo if no release version is defined (this is the default behaviour), or the “real” repo if you pass in a release version.
  4. This will also sign the release.

Note: The current assumption is that four jars will be published per project, e.g:

mongo-java-driver-3.0.0-all.jar mongo-java-driver-3.0.0-javadoc.jar mongo-java-driver-3.0.0-sources.jar mongo-java-driver-3.0.0.jar

Requirements

Because this talks to nexus, you need: ~/.gradle/gradle.properties sonatypeUsername=mongodb sonatypePassword=[find this in the settings file]

Needs a release version defined, it will do a check to make sure you have this. E.g.

./gradlew publish -DreleaseVersion=3.0.0

If you do not define a releaseVersion, it will assume you want to publish the SNAPSHOT.

Follow the steps to set up GPG/PGP

Code

We’re using a number of plugins to perform this publish process, and it’s all configured in publish.gradle. Plugins used: maven-publish for defining the publishing process. This creates the correct pom and is responsible for publishing to appropriate repo signing signs the release com.github.johnrengelman.shadow builds our uber jars

I know generally we steer clear of comments in our code, but I’ve tried to comment the publish file as much as I can, as it’s not always obvious what’s going on.

There are also specific configuration details in the build.gradle files for each project. For example, driver/build.gradle defines additional values for the pom.xml, and the correct artifact ID.

Possible errors

  • Incorrect artifact ID (i.e. the ID for sonatype to identify our jars). This is currently defined in individual project build.gradle files, in publishing.publications.mavenJava(MavenPublication).artifactId. If it’s missing, it’ll use the project name (e.g. bson).
  • Uber jar might contain too many dependencies or not enough. As per the documentation, Configure it with shadowJar { dependencies { } }
  • Nexus authentication might fail. Ensure the credentials are in ~/.gradle/gradle.properties
  • Signing might fail. Ensure you follow the steps to set up GPG/PGP
  • Could get confused between snapshot repo and real repo. publish.gradle currently assumes that if you passed in releaseVersion, you’re publish to staging. See here for more details on why we need an if statement here.
  • pom.xml might be incorrect. As per the documentation, this is configured via publishing { publications { mavenJava(MavenPublication) { pom.withXml { } } } }
  • manifest.mf is incorrect. As per the documentation), his is configured in each individual build.gradle file: jar { manifest.attributes( }

  • Wrong jar files built/uploaded. This is configured, as per the documentation, by declaring the artifacts to publish in

    publishing { publications { mavenJava(MavenPublication) { artifact sourceJar artifact javadocJar artifact shadowJar from components.java } } }

    I’m unclear where components.java comes from, but it’s the magic incantation that uploads the default jar file for a project. The Uber jar is created by applying the shadow plugin, and is included via shadowJar. The sources and javadoc jars are tasks that are defined elsewhere in the build file, for example: task sourceJar(type: Jar) { classifier = ‘sources’ from sourceSets.main.allSource }

  • Jar file name is incorrect. This is defined via the Jar plugin, and in our case we configure overrides via module build.gradle files, e.g.: archivesBaseName = ‘mongo-java-driver’

Rollback

  • a clean will remove any built artifacts, including pom.xml and manifest files.
  • published artifacts will need to be removed from Sonatype Nexus

draftReleaseNotes Creates a skeleton set of release notes. Although this could run perfectly well alone, without any of the other tasks, it does upload the uberjar so if that hasn’t previously been built you’ll get a file not found exception.

  1. Creates the skeleton of the notes content
  2. Pushes to github, so it should be visible as a draft
  3. Uploads the driver module uber jar (so just a single jar file of all classes) to the release notes. If the release tag has been successful, the release notes will also have a tag, a commit, and zips of the source files.

Note: these release notes are draft only, and can only be seen by those with the right permissions. When the release is live, the release notes will need to be published.

Requirements

Needs your github credentials in: ~/.github either login=[user name] password=[password]

                             or, if you're using two-factor auth,
                                 ~/.github
                                 oauth=[github access token]

Uses the release version you give as an argument, e.g. ./gradlew draftReleaseNotes -DreleaseVersion=3.0.0

Code

The behaviour of this task is defined in DraftReleaseNotesTask. This uses the Github API for Java, it’s this code that requires the credentials in your .github file.

Possible errors

  • Missing jar file to upload. This task does not explictly depend on archives creation, so if shadowJar or jar have not been run prior to running this task, you might get a file not found exception. The release notes will probably still have been created though.
  • Incorrect jar files attached. The jar files to attach are configured in release.gradle with the jarFile property. You can declare multiple files here. Use the task name for the task that creates the required artifact, e.g. jarFile project(’:driver’).shadowJar
  • Missing source zips, release tag and/or commit. Possibly the release tag did not get pushed correctly to https://github.com/mongodb/mongo-java-driver/.
    Ensure there’s a r3.0.0 tag or similar for the release version you’re writing the notes for.
  • Release notes are not public. Actually, this is not an error - in order to make the notes publicly visible, there’s a manual step to publish them.

Rollback

  • Go to the releases page and delete any unwanted draft release notes.

updateToNextVersion Similar to, but simpler than, prepareRelease - this task gets the code in the right state for the next working version.

  1. Update the version number in the appropriate files. In the case of the Java driver, this is build.gradle and Mongo.java. The release versions in these files are incremented (e.g. 3.0.0 to 3.0.1) and the ‘-SNAPSHOT’ added.
  2. Check all changed files into git. You should make sure you don’t have any changes outstanding before running this, as they’ll get checked in as well. The commit comment will be “Updated to next development version: ${newVersion}“, e.g. “Updated to next development version: 3.0.1-SNAPSHOT”
  3. Push the changes to the remote. This seems to push to the default remote, so if you’re working on your own fork, this pushes to your origin.

Requirements

Because this task pushes to a git remote, you’ll need github credentials: ~/.gradle/gradle.properties github.credentials.username= github.credentials.password=

Uses the release version you give as an argument, e.g. ./gradlew updateToNextVersion -DreleaseVersion=3.0.0

Code

This task is defined in UpdateToNextVersionTask. It uses JGit for the git integration.

Possible errors

  • Problems authenticating to github. [link to github troubleshooting required]
  • In theory you can run this multiple times without error - it will keep incrementing the version number (and suffixing ‘-SNAPSHOT’ to it). However, if you specify the same releaseVersion parameter every time, it should error telling you that the version doesn’t match the version in the build file (as the build file has already been incremented). This is fine, just do a manual rollback before trying again.

Rollback

  • Git: hard reset to the commit you were on before calling this task.

In Summary

There is a magic incantation which is supposed to do the full release. But if any of the steps go wrong, you can call the individual tasks alone, and follow the above guide for troubleshooting or rolling back.

Things that are open to improvement:

(AKA let’s not call this a TODO list):

  • The release plugin stuff in buildSrc is almost definitely not idiotmatic Groovy/Gradle. Instead, I’ve tried to make it readable.
  • I think I’d prefer, on reflection, to move all the stuff in buildSrc into gradle scripts instead, although that goes against the recommendations, because at least then all the gradle logic would be in the same place.
  • I’ve been encouraged to look at the gradle-git plugin instead of JGit, but I haven’t had the time.
